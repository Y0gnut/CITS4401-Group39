### 4. Stakeholder: Researchers

#### 4.1 Stakeholder Description
Researchers (including Honours, Masters, and PhD students, as well as academic staff) are the primary initiators of research activities. They are responsible for preparing and submitting ethics applications, including research methodologies, participant information, and supporting documentation. Throughout the process, they may need to respond to feedback, revise their applications, and monitor progress until a final decision is reached.

**Responsibilities:**
* Prepare and submit comprehensive ethics applications and all required supporting documents (e.g., consent forms, risk assessments).
* Provide precise research descriptions, methodologies, and participant information for committee evaluation.
* Respond to clarification requests and formally submit revised applications when required by the ethics committee.
* Monitor the real-time status and lifecycle progression of submitted applications within the system.
* Ensure all submitted research information strictly complies with institutional and ethical standards.

**Value/Benefits Expected from the System:**
* **Real-time Status Tracking**: Researchers can monitor their application progress through a centralised dashboard. This removes the need for manual email follow-ups, as the system provides live updates on workflow stages as they happen.
* **Centralised Source of Truth:** The system acts as a single integrated repository for all submissions. This centralises files and communications, which helps avoid the version confusion often caused by email chains.
* **Integrity of Document Versioning:** The system uses a versioning engine to auto-archive old drafts and mark the ‘Current Version’. This ensures reviewers always access the correct files and makes it easier to track document changes.

#### 4.2 User Stories — Researchers
* **User Story 1:** As a researcher, I want to upload and update supporting documents in the central repository. The system should automatically timestamp these files and highlight the current version, so that the committee always reviews the correct documentation.
* **User Story 2:** As a researcher, I want to view the real-time status of my application on a personal dashboard, where the system displays the current workflow stage (e.g., submitted, under review, revision required), so that I have complete visibility into my application’s progress without manual follow-up.
* **User Story 3:** As a researcher, I want to receive structured feedback through the system and submit revised versions within the same interface, and saves all changes in a revision history. This way, all feedback and updates are formally documented and easy to find.
* **User Story 4:** As a researcher, I want to record the roles of all team members within the application metadata, where the system captures these details for formal review, so that institutional compliance regarding research team responsibilities is fully satisfied.

### 5. Stakeholder: Administrative Staff

#### 5.1 Stakeholder Description
Administrative staff act as coordinators of the ethics review process. They are responsible for performing initial completeness checks, tracking application progress, managing communication between stakeholders, and ensuring the workflow operates efficiently. They play a central role in maintaining oversight of all active applications and ensuring that processes are consistent and compliant with institutional requirements.

**Responsibilities:**
* Perform rigorous completeness checks on newly submitted applications and all associated supporting documentation.
* Proactively track, update, and manage application statuses across the entire review lifecycle.
* Facilitate professional communication flows between researchers and ethics committee members.
* Coordinate the overall workflow, including routing applications for review and forwarding final outcomes.
* Maintain detailed, searchable records to ensure absolute compliance with institutional ethical policies.

**Value/Benefits Expected from the System:**
* **Unified Workflow Oversight:** The system provides an aggregated master dashboard to show all active applications, reviewer assignments, and deadlines. This allows staff to monitor the review process effectively and keep tasks on track.
* **Workflow Automation and Standardization:** By utilizing automated validation logic and digital completeness checklists, the system enforces process consistency, ensuring no application proceeds to committee review until it meets established requirements.
* **Immutable Auditability:** By logging every interaction—including timestamped status changes, document uploads, and revision requests—within a secure, system-wide database, the system ensures total transparency and simplifies compliance reporting.

#### 5.2 User Stories — Administrative Staff
* **User Story 1:** As an administrative staff member, I want to use a digital checklist to verify applications, where the system validates the presence of mandatory documents before allowing a submission to proceed, so that I can ensure only high-quality, complete applications enter the review queue.
* **User Story 2:** As an administrative staff member, I want to view all active applications via a master dashboard queue, where the system categorizes applications by status, submission date, and assigned reviewer, so that I can monitor overall efficiency and prioritize urgent tasks.
* **User Story 3:** As an administrative staff member, I want to issue formal revision requests via the system’s communication module, where the system automatically updates the application status to ‘Awaiting Revision’, so that all feedback is formally logged and traceable.
* **User Story 4**: As an administrative staff member, I want to update application statuses through the system’s workflow control panel. The system triggers real-time updates and notifications to researchers. This keeps everyone informed and avoids confusion about the current status of an application.
* **User Story 5:** As an administrative staff member, I want to filter stalled or overdue applications via the system reporting tool, where the system highlights items that have exceeded expected time limits, so that I can escalate cases to ensure timely processing.
* **User Story 6:** As an administrative staff member, I want to configure user access levels within the system. The system enforces permission boundaries based on user roles, which ensures that sensitive research data is only accessed by authorized personnel, maintaining both confidentiality and data integrity.
